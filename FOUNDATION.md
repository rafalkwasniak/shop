# FOUNDATION.md

Uniwersalna podstawa współpracy przy projektach API Rafała Kwaśniaka.

**Czym jest ten plik:** zestaw stałych zasad — niezależnych od konkretnego klienta — według których pracujemy. Komunikacja, tryb pracy, konfiguracja Git/GitHub, wymagania co do dokumentacji oraz przenośne standardy implementacji.

**Po co:** żeby przy starcie nowego projektu nie trzeba było niczego dopowiadać. Nowy projekt = skopiuj ten plik do katalogu głównego repo, a asystent czyta go na starcie i od razu wie, jak pracować.

**Relacja do `CLAUDE.md`:** `FOUNDATION.md` = zasady uniwersalne (te same w każdym projekcie). `CLAUDE.md` = specyfika danego projektu (domena, stos, decyzje produktowe). W razie konfliktu specyfika projektu z `CLAUDE.md` ma pierwszeństwo dla rzeczy projektowych; zasady współpracy z `FOUNDATION.md` obowiązują zawsze.

## 1. Komunikacja

- **Z Rafałem rozmawiamy po polsku.**
- **Kod, komentarze, nazwy, commity, PR-y i dokumentacja techniczna — po angielsku.**
- **Bez emoji i bez ozdobników** w kodzie, commitach, PR-ach i dokumentacji technicznej. Czysto i rzeczowo.
- **Komentarze tylko wtedy, gdy „dlaczego" nie jest oczywiste.** Żadnych docblocków powtarzających sygnaturę metody.
- Ton: konkretnie, bez lania wody. Raportuj uczciwie — jeśli test pada, mów wprost i pokaż wynik; jeśli krok pominięty, powiedz to.

## 2. Tryb pracy

- **Najpierw omawiamy, potem robimy.** Każdą znaczącą zmianę najpierw opisz: „co i po co". Poczekaj na zgodę Rafała, dopiero wykonaj.
- **Małe kroki.** Lepiej kilka domkniętych etapów niż jeden wielki skok.
- **Sprzątanie w przebiegu, nie na później.** Placeholdery, martwy kod, śmieci po sobie usuwamy od razu — nie odkładamy „na potem".
- **Nie zgaduj w razie konfliktu.** Gdy instrukcje projektu kłócą się ze specyfikacją albo między sobą — pytaj o decyzję.
- **Bez zbędnych abstrakcji i nowych katalogów top-level** bez wyraźnego powodu. Korzystaj z istniejącej struktury projektu.

## 3. Git i GitHub

### Setup serwera (jednorazowo, na nowym serwerze/projekcie)

1. **Wygeneruj klucz SSH na serwerze:**
   ```bash
   ssh-keygen -t ed25519 -C "rafal@kwasniak.org"
   ```
2. **Dodaj klucz publiczny w GitHub** (Settings → SSH and GPG keys, albo Deploy key w repozytorium).
3. **Ustaw remote przez SSH** (`git@github.com:...`), nie przez HTTPS.

### Tożsamość commitów (ustawiana PER-REPO)

Autorem commitów jest **wyłącznie Rafał Kwaśniak `<rafal@kwasniak.org>`**.

Ustawiamy ją **lokalnie w repozytorium**, bo globalna tożsamość Git na serwerze należy do kogoś innego i **jej nie ruszamy**:

```bash
git config user.name "Rafał Kwaśniak"
git config user.email "rafal@kwasniak.org"
```

(Bez `--global`. Zawsze sprawdź `git config user.email` w nowym repo zanim zrobisz pierwszy commit.)

### Zasady commitów

- **Bez stopki i bez atrybucji generatora.** Żadnego `Co-Authored-By`, żadnej stopki typu „generated by". Commit wygląda tak, jakby napisał go Rafał — bo to jego commit.
- **Subject po angielsku, w czasie teraźniejszym, krótki** (np. `Add OTP foundation and message queue structure`).
- **Ciało commita punktowe** dla każdego nietrywialnego commita: co zrobione / po co + lista testów. Nie tylko sam subject.
- **Commit i push po zamkniętym kawałku funkcjonalności wraz z testami** — nie per plik, nie „na koniec dnia".
- **Po każdym domkniętym elemencie sam proponuję commit i push, i czekam na zgodę Rafała.** To asystent pilnuje momentu i pyta — Rafał tylko potwierdza (albo nie). Nigdy nie pushujemy z automatu, bez zgody.

### Branching

- Branch roboczy zgodnie z ustaleniem projektu (domyślnie `develop`). PR-y kierujemy do `main`.
- PR-y bez stopki o generowaniu.

## 4. Dokumentacja API (wymagana w każdym produkcie)

Każdy projekt API ma **trzy** dokumenty, które się uzupełniają:

1. **Dokumentacja generowana z kodu z możliwością testowania** — np. OpenAPI (Scramble) + UI do odpytywania endpointów. Wzbogacana PHPDoc-iem na metodach kontrolerów (podsumowanie, parametry, odpowiedzi). To żywy kontrakt API.
2. **Narracyjny przewodnik (`api-guide.html`)** — plik, który *ludzkim językiem* tłumaczy, jak API działa i jak z niego korzystać. Narracja pisana ręcznie, nie auto-generat. Odpowiada na pytanie *jak wołać* API.
3. **Mapa kodu (`code-map.html`)** — narracyjny opis **gdzie co mieszka w kodzie**, per klasa (serwisy, modele, enumy, kontrolery, form-requesty, komendy, middleware): plik/klasa + jej publiczne metody, nie każda linia. Odpowiada na pytanie *gdzie to żyje w kodzie*. Ta sama skórka co przewodnik.

**Dostępność przez URL.** Każdy dokument HTML musi być serwowany pod **stałym, działającym URL-em** — nie „plik leżący w repo". Realizujemy to **jedną dynamiczną trasą** obejmującą cały katalog dokumentów: dowolny plik `.html` wrzucony do katalogu dokumentów (np. `docs/`) jest **od razu** dostępny pod `…/docs/<nazwa>.html`, **bez** dopisywania trasy per plik. Dzięki temu zespół (też front) dorzuca własne strony bez ruszania kodu backendu. Wymogi bezpieczeństwa tej trasy:
- nazwa pliku ograniczona do bezpiecznego sluga (`[A-Za-z0-9_-]+`) — **bez** ukośników i kropek, więc nie da się wyjść poza katalog (path traversal),
- serwujemy **wyłącznie** pliki `.html` (dosłowny sufiks w trasie) — nie wystawiamy spec `.md`, configów ani innych plików,
- katalog dokumentów żyje **poza** webrootem (jedno źródło prawdy obok spec), serwowany kontrolowanie przez trasę,
- brak pliku → 404; trasa nie koliduje z auto-generowanym kontraktem (inny prefiks/sufiks).

Po zmianach w routingu/dokumentacji sprawdzamy, że żaden z adresów nie zwraca 404.

**Utrzymanie:** przy **każdej** zmianie endpointu aktualizujemy dokumentację w tym samym kroku co kod — tak samo jak testy i tłumaczenia; przy zmianie struktury klas aktualizujemy także mapę kodu. Dokumentacja nigdy nie zostaje „na potem", a jej adresy URL pozostają żywe.

## 5. Przenośne standardy implementacji

Domyślne dla projektów API (stos Laravel). Projekt może je doprecyzować w `CLAUDE.md`, ale punkt wyjścia jest tu.

### Kształt odpowiedzi API
- Standardowo: `{ success, message, data?, pagination? }`.
- `data` dodajemy **tylko gdy są dane**.
- Klucz paginacji nazywa się `pagination`.

### Pełny CRUD i spójność danych wyjściowych
- **Pełny CRUD tam, gdzie ma to sens.** Dla każdej encji domyślnie komplet: listing, pojedynczy `GET /{id}`, create, update, delete — nie tylko podzbiór akurat potrzebny pierwszemu ekranowi. Front rzadko potrzebuje mniej, niż się wydaje na starcie.
- **Jedna encja = jeden kanoniczny kształt w odpowiedziach.** Ten sam zestaw pól zwraca listing, `GET /{id}`, create i update — nigdy się nie rozjeżdżają. Payload budowany „inline" osobno w każdym miejscu to klasa błędu (kształty cicho się rozchodzą).
- **Wzorzec: jedno źródło serializacji encji** (np. statyczna, null-safe metoda typu `card()` na modelu); serwisy i kontrolery do niej delegują. Encja zagnieżdżona w cudzym payloadzie używa tej samej metody i niesie pola renderowalne inline (nazwa, dane kontaktowe…), żeby front nie dociągał `GET /{id}` per wiersz. Zakres pól karty ustalamy z klientem.
- **Zmiana kształtu zwrotki to zmiana kontraktu** — wprowadzamy ją **addytywnie** (dodajemy pola, nie usuwamy), zgłaszamy frontowi i pokrywamy testem zamykającym kontrakt.

### Walidacja i kontrolery
- Walidacja **wyłącznie w Form Requestach**. Bez inline'owego `$request->validate()` w kontrolerach.
- Kontrolery cienkie — logikę delegujemy do serwisów.

### Konfiguracja i środowisko
- **Strojalne stałe biznesowe trzymamy w `config/`** — nie jako `const` w klasie, nie w `.env`.
- **Parytet `.env.example` z `.env`.** Każda zmiana w `.env` = aktualizacja `.env.example` w tym samym kroku.
- **`APP_DEBUG=false` na produkcji.** Diagnostyka nigdy nie trafia do odpowiedzi API.

### Logowanie i alerty
- **Logi dzienne** — rotacja per dzień (osobny plik na każdy dzień), nie jeden narastający plik.
- **Kanał Discord (webhook) do alertów/diagnostyki** — w każdym projekcie jest kanał Discord, na który idą błędy i alerty. Trzeba skonfigurować webhook i podpiąć jego obsługę (obok logów w `storage/logs`). To stały element, nie opcja.

### HTTP i routing
- **Edycja zawsze przez POST** (create i edit oba na POST; edycję rozróżnia ID w URL, np. `POST /res/{id}`). Odporne na przyszły upload pliku.
- **Upload pliku tylko przez POST** — PHP nie parsuje multipartu dla PATCH/PUT (testy tego nie wyłapią).

### Lokalizacja i testy
- Stringi kierowane do użytkownika przez warstwę tłumaczeń; **oba locale trzymamy w synchronizacji** przy dodawaniu kluczy.
- Piszemy **realne testy** (feature + unit), nie zostawiamy placeholderów ze skeletonu.
